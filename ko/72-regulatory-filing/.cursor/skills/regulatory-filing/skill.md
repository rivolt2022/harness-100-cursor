# Regulatory Filing — 인허가·신고서류 풀 파이프라인

인허가·신고의 요건조사→구비서류목록→신청서작성→첨부자료준비→제출체크리스트를 에이전트 팀이 협업하여 한 번에 생성한다.

## 실행 모드

**에이전트 팀** — 5명이 `Task` 툴로 Subagent를 호출해 협업하고 산출물을 교차 검증한다.

## Cursor 네이티브 오케스트레이션 메모

- 전문 작업은 `Task` 툴로 Subagent를 호출해 위임한다.
- 독립 브랜치는 여러 `Task` 호출을 한 번에 실행해 병렬 처리한다.
- 진행 및 의존성은 `TodoWrite`로 추적하고 산출물은 `_workspace/`에 정리한다.
- 외부 또는 실시간 시스템은 수동 웹 작업보다 MCP 도구/리소스를 우선 사용한다.
## 에이전트 구성

| 에이전트 | 파일 | 역할 | 타입 |
|---------|------|------|------|
| requirements-investigator | `.cursor/agents/requirements-investigator.md` | 법령·요건 조사, 구비서류 목록 | general-purpose |
| document-drafter | `.cursor/agents/document-drafter.md` | 신청서·사업계획서·양식 작성 | general-purpose |
| attachment-preparer | `.cursor/agents/attachment-preparer.md` | 첨부자료 준비 가이드, 정합성 확인 | general-purpose |
| submission-verifier | `.cursor/agents/submission-verifier.md` | 최종 검증, 제출 체크리스트 | general-purpose |

## 워크플로우

### Phase 1: 준비 (오케스트레이터 직접 수행)

1. 사용자 입력에서 추출한다:
    - **인허가 유형**: 어떤 허가/등록/신고를 받으려 하는가
    - **업종/사업 내용**: 구체적인 사업 분야
    - **사업장 정보** (선택): 소재지, 면적, 시설 현황
    - **신청인 정보** (선택): 개인/법인, 자격, 경력
    - **기존 서류** (선택): 이미 준비된 서류
2. `_workspace/` 디렉토리를 프로젝트 루트에 생성한다
3. 입력을 정리하여 `_workspace/00_input.md`에 저장한다
4. 기존 서류가 있으면 `_workspace/`에 복사하고 해당 Phase를 건너뛴다
5. 요청 범위에 따라 **실행 모드를 결정**한다 (아래 "작업 규모별 모드" 참조)

### Phase 2: 팀 구성 및 실행

| 순서 | 작업 | 담당 | 의존 | 산출물 |
|------|------|------|------|--------|
| 1 | 요건 조사·구비서류 목록 | investigator | 없음 | `_workspace/01_requirements_research.md`, `_workspace/02_document_list.md` |
| 2a | 신청서·양식 작성 | drafter | 작업 1 | `_workspace/03_application_draft.md` |
| 2b | 첨부자료 준비 가이드 | preparer | 작업 1 | `_workspace/04_attachments_guide.md` |
| 3 | 제출 검증·체크리스트 | verifier | 작업 2a, 2b | `_workspace/05_submission_checklist.md` |

작업 2a(신청서 작성)와 2b(첨부자료 준비)는 **병렬 실행**한다.

**팀원 간 소통 흐름:**
- investigator 완료 → drafter에게 서식 종류·기재 주의사항 전달, preparer에게 구비서류 목록·발급처 전달
- drafter 완료 → preparer에게 신청서 기재 수치·명칭 전달 (정합성 확인), verifier에게 작성 완료 서류 전달
- preparer 완료 → verifier에게 자료 준비 상태·정합성 점검 결과 전달
- verifier는 모든 산출물을 교차 검증. 🔴 접수 거부 사유 발견 시 해당 에이전트에게 수정 요청 → 재작업 → 재검증 (최대 2회)

### Phase 3: 통합 및 최종 산출물

1. `_workspace/` 내 모든 파일을 확인한다
2. 검증 보고서의 🔴 접수 거부 사유가 모두 해소되었는지 확인한다
3. 최종 요약을 사용자에게 보고한다:
    - 요건 조사 보고서 — `01_requirements_research.md`
    - 구비서류 목록 — `02_document_list.md`
    - 신청서 초안 — `03_application_draft.md`
    - 첨부자료 가이드 — `04_attachments_guide.md`
    - 제출 체크리스트 — `05_submission_checklist.md`

## 작업 규모별 모드

| 사용자 요청 패턴 | 실행 모드 | 투입 에이전트 |
|----------------|----------|-------------|
| "인허가 서류 전체 준비해줘" | **풀 파이프라인** | 4명 전원 |
| "어떤 서류가 필요한지만 알려줘" | **요건조사 모드** | investigator 단독 |
| "이 신청서 검토해줘" (기존 파일) | **검증 모드** | verifier 단독 |
| "신청서만 작성해줘" (요건 파악 완료) | **작성 모드** | drafter + verifier |
| "첨부서류 준비 방법 알려줘" | **자료준비 모드** | investigator + preparer |

**기존 파일 활용**: 사용자가 기존 서류를 제공하면, 해당 파일을 `_workspace/`의 적절한 위치에 복사하고 해당 단계의 에이전트는 건너뛴다.

| 전략 | 방식 | 용도 |
|------|------|------|
| 파일 기반 | `_workspace/` 디렉토리 | 주요 산출물 저장 및 공유 |
| 메시지 기반 | Task 툴로 Subagent 호출 | 실시간 핵심 정보 전달, 수정 요청 |

## 에러 핸들링

| 에러 유형 | 전략 |
|----------|------|
| 웹 검색 실패 | 일반 법령 지식 기반 작업, 보고서에 "최신 법령 확인 필요" 명시 |
| 법령 불명확 | 복수 해석 가능성을 병기, 관할기관 사전상담 권고 |
| 사용자 정보 부족 | 플레이스홀더 삽입 + 기입 가이드, 미기입 항목 요약 제공 |
| 에이전트 실패 | 1회 재시도 → 실패 시 해당 산출물 없이 진행, 체크리스트에 누락 명시 |
| 검증에서 🔴 발견 | 해당 에이전트에 수정 요청 → 재작업 → 재검증 (최대 2회) |

## 테스트 시나리오

### 정상 흐름
**프롬프트**: "서울에서 일반음식점 영업허가 받으려면 어떤 서류가 필요하고 어떻게 준비해야 하는지 전체 서류를 만들어줘"
**기대 결과**:
- 요건 조사: 식품위생법 근거, 관할 구청 보건위생과, 인적·물적 요건 정리
- 구비서류: 영업허가신청서, 위생교육이수증, 건축물대장, 수질검사성적서 등 목록
- 신청서: 별지 서식에 따른 신청서 초안 + 사용자 기입 항목 안내
- 첨부자료: 각 서류 발급처·비용·소요시간·유효기간 안내
- 체크리스트: 전체 서류 완전성·정합성 검증 + 제출 순서

### 기존 파일 활용 흐름
**프롬프트**: "이 신청서 초안 검토해주고 빠진 서류 있으면 알려줘" + 신청서 파일 첨부
**기대 결과**:
- 기존 신청서를 `_workspace/03_application_draft.md`로 복사
- 검증 모드: investigator + verifier 투입
- drafter, preparer는 건너뜀 (필요 시 verifier가 보완 요청)

### 에러 흐름
**프롬프트**: "화장품 제조업 등록하려는데 서류 준비해줘, 아직 사업장 계약 전이야"
**기대 결과**:
- 물적 요건(시설 기준) 조사 후 "사업장 확보 전 사전 준비 가능 서류" vs "사업장 확보 후 준비 서류" 구분
- 사업장 관련 항목은 플레이스홀더 처리
- 체크리스트에 "사업장 확보 후 보완 필요 항목" 별도 섹션
